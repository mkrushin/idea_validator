# Технический план: фича «План валидации»

**Статус:** черновик спеки, ждёт одобрения
**Дата:** 2026-07-14
**Вводные (согласованы):** генерация — гибрид (шаблон + AI); оплата — Lemon Squeezy; порядок — сначала бесплатный генератор, платный gate вторым шагом.

---

## 1. Зачем это

Все конкуренты (ValidMyIdea, IdeaProof) продают **отчёт об идее** и бросают пользователя. Методология валидации (статья Pennington) прямо говорит: решает не отчёт, а **прямые разговоры с людьми и «10–40 неспровоцированных да»**. Никто не доводит до этого шага.

«План валидации» — это мост от отчёта к действию: **не «что мы думаем о твоей идее», а «вот что тебе сделать завтра, чтобы проверить её руками»**. Это и уникальное позиционирование, и будущий платный слой.

---

## 2. Что именно генерируем (содержание плана)

7 блоков, опираются на уже готовый анализ идеи (раздел 2 «Target Customer» и раздел 3 «Pain Points» — источник для портрета и каналов):

| # | Блок | Источник данных |
|---|------|-----------------|
| 1 | **Кого спросить** — портрет 1–2 сегментов, где у них болит | из раздела 2 анализа |
| 2 | **Где их найти** — 3–5 конкретных каналов (сабреддиты, сообщества, чаты, оффлайн) под нишу | AI-подстановка по нише |
| 3 | **5 вопросов для интервью** — в стиле The Mom Test: про прошлое поведение, не про «нравится ли идея» | шаблон + AI-адаптация под идею |
| 4 | **Шаблон холодного сообщения** — короткий текст для DM/почты, чтобы позвать на разговор | шаблон + подстановка |
| 5 | **Гипотеза мини-лендинга** — 1 заголовок + 1 CTA (waitlist/предзаказ) для проверки спроса трафиком | AI по идее |
| 6 | **Пороги go/stop** — сколько «да» = сигнал (10–40 unsolicited yeses; ~5% конверсия лендинга) | фикс-шаблон |
| 7 | **Чеклист-трекер** — 5–7 шагов с чекбоксами (поговорил с N, собрал лендинг, получил X сигапов…) | фикс-шаблон |

**Гибрид на практике:** блоки 3, 6, 7 — фиксированный каркас (работает всегда, включая demo). Блоки 1, 2, 4, 5 — AI подставляет конкретику. Если AI недоступен (demo/нет баланса) → отдаём каркас с плейсхолдерами вместо AI-частей. Пользователь всегда получает рабочий результат.

## 3. Этап 1 — Бесплатный генератор (делаем первым)

### 3.1. Данные
`ALTER TABLE ideas ADD COLUMN validation_plan TEXT;` — план кэшируется на идею, чтобы не генерить повторно.

Миграция в `init_db()` идемпотентно:
```python
# после CREATE TABLE ideas
cols = [r[1] for r in cursor.execute("PRAGMA table_info(ideas)").fetchall()]
if "validation_plan" not in cols:
    cursor.execute("ALTER TABLE ideas ADD COLUMN validation_plan TEXT")
```

### 3.2. Генерация
Новый файл `content/validation_plan_prompt.txt` — гибридный промпт (жёсткая структура из §2, AI заполняет только блоки 1/2/4/5, формат тот же `###`-разбиение, что и анализ, чтобы переиспользовать фронтовый `displayAnalysis`).

Функция в `backend/claude_api.py`:
```python
def generate_validation_plan(idea_text: str, analysis_text: str) -> str:
    if not client:
        return get_demo_validation_plan(idea_text)   # каркас-fallback
    prompt = load_prompt("validation_plan_prompt.txt").format(
        idea=idea_text, analysis=analysis_text
    )
    msg = client.messages.create(model="claude-3-5-sonnet-20241022",
                                 max_tokens=1500, messages=[{"role":"user","content":prompt}])
    return msg.content[0].text
```
(`load_prompt_template` обобщить до `load_prompt(filename)`.)

### 3.3. API
`POST /api/validation-plan` — тело `{ "idea_id": int }`:
1. `get_idea(idea_id)` → если нет → 404.
2. Если `validation_plan` уже есть в БД → вернуть его (кэш).
3. Иначе `generate_validation_plan(idea_text, analysis)` → сохранить в БД → вернуть.

Модель ответа `ValidationPlanResponse { idea_id, plan }`.

### 3.4. Frontend
Под карточкой анализа — новая карточка «План действий: как проверить идею» с кнопкой **«Составить план валидации»**. По клику → `POST /api/validation-plan` → рендер тем же `markdownToHtml`/`displayAnalysis` (переиспользуем стили секций и нумерацию 01–07).

Чеклист (блок 7) — интерактивные `<input type=checkbox>`, состояние в `localStorage` по `idea_id` (без аккаунтов; MVP-достаточно).

### 3.5. Объём Этапа 1
- 1 миграция, 1 промпт-файл, 1 функция + demo-fallback, 1 endpoint, 1 модель, 1 карточка на фронте.
- **Оценка: ~0.5–1 день.** Реального AI не требует (работает на каркасе в demo), деньги не нужны.

## 4. Этап 2 — Платный gate (Lemon Squeezy, вторым шагом)

**Важно:** базовый план из Этапа 1 остаётся бесплатным (он собирает спрос и фидбек). За деньги уводим **надстройку**, а не прячем базу за пейволл. Кандидаты на платное (выбрать 1–2 при запуске этапа):
- Экспорт плана в **PDF/Notion** (аккуратный документ, который не стыдно сохранить).
- **Углублённый разбор одного канала** (например, детальный сценарий для конкретного сабреддита + готовые посты).
- **Повторные генерации** сверх 1 бесплатной.

### 4.1. Lemon Squeezy
- Продукт: разовая покупка $5–9, тип — с выдачей **license key** (LS умеет генерить ключи на заказ).
- Почему license key, а не аккаунты: у продукта **нет авторизации**. Ключ решает привязку без строительства аккаунтов — пользователь оплачивает → получает ключ от LS на email → вводит его в приложении → разблокировка.

### 4.2. Данные
```sql
CREATE TABLE IF NOT EXISTS unlocks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    license_key TEXT UNIQUE NOT NULL,
    email TEXT,
    ls_order_id TEXT,
    status TEXT DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.3. Поток
1. Кнопка «Разблокировать PDF/расширенный план» → редирект на LS checkout link.
2. LS webhook `POST /api/ls-webhook` (событие `order_created`) → проверить подпись `X-Signature` (HMAC по `LEMONSQUEEZY_WEBHOOK_SECRET`) → сохранить `license_key` в `unlocks`.
3. Пользователь вводит ключ → `POST /api/unlock {license_key}` → валидируем через LS API `/v1/licenses/validate` (или по своей таблице) → выдаём доступ (флаг в `localStorage` + серверная проверка при генерации платной части).

### 4.4. Инфраструктура — ПРОВЕРЕНО ✓
Volume на Railway примонтирован: `idea_validator-volume` → `/app/data` (33 MB / 500 MB). SQLite-данные (отзывы, будущие покупки) переживают редеплой. Осталось убедиться, что `DATABASE_PATH` в переменных сервиса указывает внутрь `/app/data` (иначе БД пишется мимо Volume).

### 4.5. Объём Этапа 2
- Настройка LS (продукт, webhook, ключи) + таблица `unlocks` + webhook-endpoint + валидация ключа + PDF-генерация (напр. `weasyprint`/готовый HTML→PDF).
- **Оценка: ~1–2 дня** + время на верификацию Volume и тест webhook.
