# Idea Validator — Техническая спецификация

> **Этот документ становится рабочим только после прохождения Gate 0→1** (см. `03_ROADMAP.md`). На Этапе 0 кода нет — вообще. Если ты Claude Code и читаешь это в начале работы над проектом — сначала проверь в `STATUS.md`, пройден ли gate.

---

## 0. Принципы реализации

1. **Продукт — пайплайн + отчёт, а не веб-приложение.** Никакого клиентского кабинета, auth и БД-инфраструктуры до Этапа 2, и то по доказанной необходимости. Это главное отличие от техспеки BuildLoop: там социальный продукт тянул за собой Next.js/Supabase, здесь ядро — Python-скрипты, играющие на сильную сторону автора.
2. **Переиспользовать уже построенное.** Инструменты постинга/мониторинга: `/Users/alexm/vc/reddit`, `/telegram` (проверены вживую), `/twitter`, `/threads` (построены, не доведены). Образец CRM на json + скриптах: `buildloop-landing/crm/`. Опыт лендинга: `buildloop-landing` (Next.js + Vercel + Formspree).
3. **Ручная курация — принципиальная позиция, не техдолг.** Каждый пост правится и публикуется человеком. Автоматизируем черновики, мониторинг, учёт — но не решение «что и куда постить». Никаких мультиаккаунтов, anti-detection, обхода капчи и рейт-лимитов.
4. **Мерить с первого заказа.** Часы по стадиям, ответы, возвраты — в CRM с Этапа 0 (вручную), скриптами — с Этапа 1. Это данные для gate'ов.
5. **Python3 везде, где выбор за нами.** Зависимости через pip.

## 1. Инструментарий по этапам

### Этап 0 — Консьерж (ноль кода продукта)

| Задача | Инструмент | Комментарий |
|--------|-----------|-------------|
| Бриф идеи | Переписка / созвон + шаблон вопросов (markdown в `content/`), включая блок согласия на объём раскрытия идеи | Шаблон брифа — единственный «артефакт» этапа |
| AI-анализ | Claude (чат или Claude Code) + промпт по мотивам 9 шагов ShipFit | Промпт сохранить в `content/prompts/` — он станет ядром пайплайна Этапа 1 |
| Черновики постов | Тот же промпт-набор + скилл humanize | Публикация — руками |
| Постинг и мониторинг | Существующие скрипты `/Users/alexm/vc/reddit`, `/telegram` как помощники | Каждый пост — вручную, с проверкой правил площадки |
| Опрос панели | PickFu через веб-интерфейс | Наличие/пригодность API не проверялось — на этом этапе и не нужно |
| Отчёт клиенту | Markdown → PDF (pandoc) или Notion-страница | Шаблон отчёта сохранить в `content/templates/` |
| Оплата | Stripe payment link / PayPal-инвойс — что проще завести | 2–3 транзакции; MoR-вопросы (VAT) отложены до Этапа 1 |
| Учёт | `crm/data/orders.json` руками или таблица | Часы по стадиям писать сразу, не «потом вспомню» |

### Этап 1 — Продуктизация (минимальный код)

Три компонента, в порядке сборки:

**1. CRM заказов** (по образцу `buildloop-landing/crm/`): `crm/data/orders.json` + скрипты `add_order.py` / `update_order.py` / `report.py` + тесты. Модель данных — §2. Отчёт скрипта = прогресс к gate'ам (заказы, возвраты, медиана часов, доля холодных).

**2. Пайплайн анализа** (`pipeline/`):
- `analyze.py` — бриф (markdown/json) → черновик анализа по 8 разделам (Claude API, модель уровня Sonnet; себестоимость копеечная — см. бизнес-план).
- `draft_posts.py` — идея + целевые площадки → черновики постов под культуру каждой площадки.
- `build_report.py` — анализ + сведённые живые ответы → итоговый отчёт (markdown → PDF).
- Всё — CLI-скрипты с ручной правкой результата между шагами. Никакого «нажал кнопку — клиент получил отчёт».

**3. Лендинг + приём заказов:**
- Статический лендинг (Next.js на Vercel — опыт buildloop-landing — или проще; решить по месту): оффер, пример отчёта, цена, FAQ.
- Оплата: payment link (Stripe) или MoR-провайдер (Lemon Squeezy / Paddle — сами считают EU VAT, что для основателя в ЕС снимает головную боль; сравнить комиссии на реальном объёме — на 5–10 заказах/мес разница непринципиальна).
- Бриф: форма (Tally / Formspree — второй уже настроен в buildloop-landing) → письмо/вебхук → `add_order.py`.
- Никакого auth и личного кабинета.

### Этап 2 — Надёжность канала

- `sources/playbook.json` — база площадок: категория идеи → площадки, правила, лимиты, история результатов. Наполняется из заказов.
- Мониторинг ответов: расширение существующих `fetch_replies`-скриптов под привязку к заказу (`order_id` в tracked-записях). Известное ограничение из опыта BuildLoop: ответы без корректного reply_to теряются — закладывать ручную сверку.
- Статусы клиенту: email-апдейты (вручную или простой скрипт) — страница статуса только если клиенты явно просят.
- Учёт здоровья канала в CRM: удаления постов, предупреждения, баны — поля в `posts`, отчёт в `report.py`.

### Этап 3 — Масштаб

Не специфицируется заранее. Маркетплейс ревьюеров, кабинет клиента, автоматизация делегирования — отдельная спецификация, когда (если) Gate 2→3 пройден и данные покажут, какая из опций Этапа 3 выбрана.

## 2. Модель данных (CRM, Этап 1)

> Формат — json-файлы + скрипты (как в buildloop CRM), не БД. Объёмы (десятки заказов) не оправдывают ничего тяжелее. Переезд на SQLite — только если json станет узким местом на практике.

### orders.json — заказ
| Поле | Тип | Заметки |
|------|-----|---------|
| id | str | `ord-001`, сквозная нумерация |
| client | obj | имя/ник, контакт, источник (площадка/личка/лендинг) |
| idea_brief | str | путь к файлу брифа в `orders/ord-001/` |
| price_usd | int | фактическая цена (на тесте — разные точки) |
| tier | str | `concierge` на Этапах 0–1; тарифы позже |
| status | enum | `lead → paid → in_progress → delivered → closed / refunded` |
| cold | bool | пришёл без личной продажи (метрика gate 1→2) |
| hours | obj | `{brief, analysis, posts, monitoring, report}` — часы по стадиям |
| panel | obj/null | `{provider, responses, cost_usd}` |
| feedback | obj/null | самая ценная часть отчёта, купил бы снова, цитата |
| dates | obj | paid_at, delivered_at |

### posts (внутри заказа) — публикация живого блока
| Поле | Тип | Заметки |
|------|-----|---------|
| platform | str | reddit / telegram / ... |
| target | str | сабреддит / чат |
| url | str | |
| disclosure | str | формат раскрытия, использованный в посте (тестируем на Этапе 0) |
| status | enum | `draft → published → removed / warned / ok` — здоровье канала |
| substantive_replies | int | содержательные ответы (критерий — как в buildloop CRM) |

### Отчёт `report.py` должен считать
Прогресс к текущему gate: платные заказы (всего / холодные), возвраты %, медиана часов на заказ, медиана содержательных ответов, здоровье канала (удаления/баны на 10 заказов), доля клиентов, назвавших живой блок самым ценным.

## 3. Структура проекта (целевая, с Этапа 1)

```
idea-validator/
├── IDEA.md                    # исходный документ идеи (первоисточник фактов)
├── docs/                      # этот пакет документации + STATUS.md
├── content/
│   ├── prompts/               # промпты анализа и черновиков постов (появляются на Этапе 0)
│   ├── templates/             # шаблоны брифа и отчёта (Этап 0)
│   └── offers/                # тексты офферов по площадкам
├── crm/
│   ├── data/orders.json
│   └── scripts/               # add_order / update_order / report + тесты
├── orders/                    # рабочие файлы заказов: ord-001/{brief,analysis,report}.md
├── pipeline/                  # analyze.py, draft_posts.py, build_report.py (Этап 1)
└── landing/                   # если лендинг живёт в этом репо (решить на Этапе 1)
```

На Этапе 0 из этого существуют только `IDEA.md`, `docs/`, `content/` и `orders/` с ручными файлами.

## 5. Чего НЕ реализовывать (стоп-границы для кода)

- **Ничего до Gate 0→1.** Весь Этап 0 — руками.
- **Массовый автопостинг, мультиаккаунты, anti-detection, обход капчи/рейт-лимитов** — ни на каком этапе. Это и позиция продукта, и требование площадок.
- **Веб-приложение с auth/кабинетом** — не раньше Этапа 2, и только по явным запросам клиентов.
- **Маркетплейс ревьюеров** — не раньше Этапа 3 (отдельная спецификация).
- **Лендинг-генератор для клиентов** — не строить (коммодити; максимум — ручной смоук-лендинг в премиум-тарифе, если спрос покажет).
- **Свой биллинг** — только payment links / MoR-провайдер.
- **Интеграции с панелями по API** — пока объём не оправдает; веб-интерфейс PickFu достаточен (наличие API не проверялось — проверить, когда дойдёт до дела).
